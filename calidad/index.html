Calidad del software en History Web

Este apartado resume cómo se aplican normas, modelos, estándares, programación, pruebas y recomendaciones al desarrollo de History Web. El objetivo es que el sitio sea una línea del tiempo clara y también un ejemplo sencillo de proyecto web con buena calidad.

Normas

Aunque History Web es un proyecto académico, se inspira en normas internacionales de ingeniería de software. De forma simplificada, tomamos ideas de:

  • ISO/IEC 12207 – Procesos del ciclo de vida del software: separamos claramente las actividades de planificación, diseño de la interfaz, codificación y verificación.
  • ISO/IEC 25010 y la antigua ISO/IEC 9126 – Modelo de calidad del producto: usamos sus características como guía (usabilidad, mantenibilidad, portabilidad, fiabilidad básica).

En la práctica, estas ideas se concretan en decisiones muy visibles dentro del sitio:

  • Uso de HTML5 semántico (<header>, <main>, <section>, <footer>) para organizar la información.
  • Accesibilidad: enlace para “Saltar al contenido principal”, textos alternativos en las imágenes y contraste adecuado entre fondo y texto.
  • Navegación coherente en todas las páginas: mismo menú fijo arriba y mismo pie de página, lo que reduce errores y confusiones.
  • Contenido estructurado en secciones: eventos principales, subeventos y páginas de detalle con el mismo patrón de diseño.

Modelos

Para evaluar la calidad de History Web usamos el modelo de ISO/IEC 25010 como referencia conceptual, aunque de forma adaptada al contexto educativo. Nos centramos sobre todo en:

  • Usabilidad: interfaz limpia, títulos claros, botones destacados (oro y rojo) y lectura cómoda en todas las secciones.
  • Mantenibilidad: estructura de carpetas simple (assets, eventos, subcarpetas por era, calidad) y estilos centralizados en styles.css.
  • Portabilidad: el sitio está preparado para funcionar en local y en GitHub Pages usando rutas relativas.
  • Fiabilidad básica: todos los enlaces se revisan para evitar errores 404 y mantener una experiencia consistente.

Estándares

History Web se apoya en estándares abiertos que favorecen la interoperabilidad y la comprensión del código:

  • HTML5 y CSS3 (W3C): se respeta la sintaxis y la estructura recomendada; los estilos se organizan con clases reutilizables como event-card, event-detail-main o subpage-main.
  • Bootstrap 5: se utiliza para la grilla responsiva, la barra de navegación y algunos espaciados, lo que simplifica el soporte a distintos tamaños de pantalla.
  • WCAG 2.1 (a nivel básico): se cuidan contraste, tamaños de fuente legibles, navegación por teclado y uso de textos descriptivos en los enlaces.
  • Buenas prácticas de código: nombres de clases descriptivos, indentación consistente y separación clara entre contenido (HTML) y presentación (CSS).

Código y programación

La implementación se centra en HTML y CSS con apoyo de Bootstrap, pero se siguen criterios de diseño típicos de proyectos más grandes:

  • Una única hoja de estilos global: assets/css/styles.css con variables de color (--gold, --red, --text-main) y componentes reutilizables (tarjetas, botones, layouts).
  • Páginas de eventos principales con tarjetas en grilla y subpáginas de detalle con título, imagen de 512×512 y texto largo organizado en secciones.
  • Rutas relativas pensadas para funcionar tanto en local como en GitHub Pages, evitando URLs absolutas que puedan romperse.
  • Estructura modular de carpetas: cada bloque histórico (origen del universo, formación de la Tierra, eras geológicas, edades históricas) tiene su propio index.html y subeventos.

Pruebas y evaluación

Las pruebas en History Web son principalmente manuales, pero se inspiran en la familia de normas ISO/IEC/IEEE 29119 sobre pruebas de software, adaptadas a un entorno web sencillo. Se consideran varios tipos de prueba:

  • Pruebas funcionales: comprobar que todos los enlaces funcionan (eventos, subeventos, botones de “Ver evento”, “Volver a eventos”, navegación entre Calidad y Acerca de).
  • Pruebas de regresión visual: cada vez que se agrega un nuevo evento se revisa que el diseño de las tarjetas siga alineado y no se rompa la distribución de la grilla.
  • Pruebas responsivas: revisar el sitio en escritorio, tableta y móvil usando las herramientas del navegador para asegurar que menús, tarjetas e imágenes se adaptan.
  • Pruebas de accesibilidad básica: uso de Lighthouse para medir contraste, uso de etiquetas correctas y facilidad de navegación por teclado.
  • Pruebas de rendimiento: se mide con Lighthouse el tiempo de carga, tamaño de imágenes y uso de recursos externos (por ejemplo, Bootstrap desde CDN).

Para mantener o mejorar estos valores, se propone el siguiente checklist de pruebas rápidas cada vez que se actualice el sitio:

  • Enlaces: comprobar que los botones “Ver evento”, “Volver a eventos” y la navegación del menú funcionen en todas las páginas nuevas.
  • Responsivo: revisar la visualización en escritorio, tableta y móvil usando las herramientas del navegador.
  • Accesibilidad: ejecutar Lighthouse y verificar que no aparezcan errores graves (contraste, etiquetas sin texto, botones sin nombre accesible).
  • Rendimiento: ejecutar Lighthouse y confirmar que las métricas de Performance se mantengan en valores altos (por encima de 90 cuando sea posible).
  • Código: validar que los nuevos archivos HTML respeten la misma estructura (navbar, main, footer) y usen las clases existentes de estilos.
  • Contenido: revisar ortografía, coherencia de fechas, nombres de eventos y descripciones históricas.

Conclusiones

History Web muestra que es posible construir una línea del tiempo educativa con un diseño atractivo y, al mismo tiempo, aplicar principios de calidad de software inspirados en normas ISO. La organización por eventos y subeventos, el uso de componentes reutilizables y la concentración de estilos en una sola hoja facilitan el mantenimiento y la evolución del proyecto.

El proyecto no pretende cubrir todos los requisitos de las normas formales, pero sí acercar al estudiante a la forma de pensar de la ingeniería de software: procesos claros, criterios de calidad y evidencia mediante pruebas.

Recomendaciones

Para futuras versiones de History Web se recomienda:

  • Añadir pruebas automáticas (por ejemplo, scripts que verifiquen enlaces rotos o validen el HTML y el CSS).
  • Documentar en un README los pasos para desplegar el proyecto y actualizar eventos, incluyendo una guía rápida de calidad.
  • Incorporar un módulo de JavaScript para hacer más interactiva la línea del tiempo (búsqueda de eventos, filtros por era, marcadores de favoritos).
  • Registrar de forma histórica los resultados de las auditorías de Lighthouse para comparar versiones del sitio y detectar mejoras.
